# CRB GA — Telegram Mini App (гарант сделок)

Гарант-сервис (эскроу) для безопасных сделок в Telegram. Приём оплаты:
**xRocket**, **Bitpapa**, **PGon**, **NicePay**, **RuKassa**.
Вход: автоматически внутри Telegram (Mini App) или через Telegram Login Widget в браузере.

Средства покупателя замораживаются у гаранта и выплачиваются продавцу только
после подтверждения приёмки. При конфликте открывается спор, который разбирает
арбитр.

## Из чего состоит

| Часть | Путь | Описание |
|---|---|---|
| Mini App (фронтенд) | `index.html`, `css/`, `js/` | Статическая страница без сборки: список сделок, создание, статусы, оплата |
| Бэкенд | `backend/` | Node.js + Express: сделки, счета xRocket Pay, webhook об оплате, заявки Bitpapa |

## Флоу сделки

1. **Создана** — стороны договорились, создана сделка с условиями и суммой.
2. **Оплачена** — покупатель заплатил (xRocket-счёт или перевод Bitpapa с кодом сделки), средства у гаранта.
3. **Выполнена** — продавец передал товар / выполнил условия.
4. **Завершена** — покупатель подтвердил приёмку, гарант выплачивает продавцу сумму за вычетом комиссии.
5. **Спор** — на шагах 2–3 любая сторона может открыть спор, его решает арбитр.

## Быстрый старт (демо-режим)

Фронтенд работает без бэкенда: сделки хранятся в localStorage, оплата
симулируется, есть кнопки «Демо: …» для прохода всего флоу.

1. Опубликуйте корень репозитория на GitHub Pages (Settings → Pages → Deploy from branch).
2. В [@BotFather](https://t.me/BotFather): `/newbot`, затем `/setmenubutton` (или Bot Settings → Menu Button) и укажите URL страницы.
3. Откройте бота → кнопка меню → Mini App запустится.

### Локальная админ-зона настроек

Для локальной настройки клиентских переменных Mini App откройте:

```text
http://127.0.0.1:3000/?admin=1
```

На localhost страница `/?admin=1` теперь сразу открывает локальную demo-admin
сессию и показывает вкладку `Админ` без дополнительного клика по входу.

Во вкладке `Админ` можно менять `API_URL`, `BOT_USERNAME`, `FEE_PERCENT`,
`SUPPORT_USERNAME`, `BITPAPA_ACCOUNT` и `PAY_METHODS`. Эти overrides
сохраняются в `localStorage` браузера и переопределяют `js/config.js`
только локально. Серверные секреты из `backend/.env` экран не изменяет.

Если вы открываете сервис локально с `localhost` / `127.0.0.1`, в админ-панели
доступен и серверный раздел: там можно задать `BOT_TOKEN` без ручного
редактирования `.env`. Токен проверяется через Telegram API и сохраняется в
`backend/runtime_settings.json`.

Если старый backend уже занимает `:3000`, а обновлённый backend поднят на
`:3001`, локальная Mini App с `API_URL: "auto"` автоматически предпочитает
API на `http://127.0.0.1:3001`.

### Расширенная админ-панель

Во вкладке `Админ` теперь доступны:

- **Runtime-настройки клиента** — `API_URL`, `BOT_USERNAME`, `FEE_PERCENT`, список платёжек;
- **Серверный BOT_TOKEN** — обновление Telegram API key бота прямо из панели;
- **Поиск по сервису** — поиск по сделкам и пользователям;
- **Статистика по пользователям** — агрегаты по ролям, спорам, завершённым сделкам и обороту;
- **Сделки и арбитраж** — список сделок, ручное подтверждение оплат и решение споров.

### Команды Telegram-бота

- `/start`, `/app` — открыть Mini App;
- `/post` — сгенерировать пост-объявление с кнопками на разные шаблоны сделок (берутся ваши сделки с ролью продавца);
- инлайн-режим `@ваш_бот <поиск>` — вставить в чат объявление по шаблону сделки с кнопкой «Оформить»;
- при нажатии на кнопку в объявлении бот создаёт новую сделку с продавцом и автозаполняет её по выбранному шаблону.

Команды гаранта (`ADMIN_IDS`):

- `/admin` — сводка и **актуальные задачи с кнопками**: заявки об оплате Bitpapa (кнопка «✅ Подтвердить оплату») и споры (кнопки «✅ Продавцу» / «🔄 Покупателю»);
- `/setup` — перенастроить профиль бота (имя, описание, команды, кнопку меню) из текущих значений;
- `/group` — привязать текущую группу как админскую;
- `/deal <ID>` — карточка сделки.

### Управление прямо из Telegram

Гаранту не нужно открывать приложение, чтобы вести сделки:

- при заявке об оплате Bitpapa бот присылает уведомление с кнопкой **«Подтвердить оплату»** — нажатие переводит сделку в «Оплачена»;
- при открытии спора приходит уведомление с кнопками **«Продавцу» / «Покупателю»** — решение сразу применяется (в пользу продавца запускается автовыплата);
- `/admin` в личке или в админ-группе выводит все открытые задачи такими же кнопками.

Кнопки действуют только для Telegram ID из `ADMIN_IDS`; после нажатия кнопки убираются, чтобы действие нельзя было повторить.

### Управление из Mini App

Пользователю из `ADMIN_IDS` в приложении появляется вкладка **«Админ»**: список сделок с фильтрами (ожидают оплату / споры / активные / все), подтверждение оплат и решение споров в один тап, сводная статистика и переход к серверным настройкам (`settings.html`). Права определяет сервер по `/api/me` — вкладка не показывается обычным пользователям.

### Админ-группа для бота

Можно привязать одну основную Telegram-группу/супергруппу для служебных уведомлений:

1. Добавьте бота в нужную админ-группу.
2. Добавьте в `ADMIN_IDS` Telegram ID администратора, который будет привязывать группу.
3. В группе выполните `/group` от имени этого администратора.

После привязки бот будет отправлять в группу уведомления (с кнопками действий) о:

- новых спорах;
- запросах поддержки;
- ручных подтверждениях/проверках оплат.

Если хотите привязать группу сразу через окружение, укажите `ADMIN_GROUP_ID` в `backend/.env`.

## Боевой режим (с бэкендом)

### Минимальный запуск: впишите ключи → работает

Достаточно заполнить `backend/.env` и запустить один процесс — он отдаёт и
Mini App, и API, и поднимает бота:

1. `BOT_TOKEN` — токен из [@BotFather](https://t.me/BotFather). Без него бот и
   авторизация не работают; с ним бот сам стартует (long-polling), выставляет
   команды, описание и кнопку меню.
2. `ADMIN_IDS` — ваш Telegram ID (узнать: [@userinfobot](https://t.me/userinfobot)).
   Даёт доступ к вкладке «Админ» и кнопкам управления в Telegram.
3. Ключи платёжек — вписывайте только те, что используете; остальные методы
   можно скрыть в `PAY_METHODS` (`js/config.js`). Приём заработает сразу после
   ввода ключей, перезапуск не требуется (ключи можно менять и из вкладки
   «Админ» → «Серверные настройки»).
4. `PUBLIC_URL` — публичный адрес бэкенда (для callback-ссылок и вебхуков).
5. `WEBHOOK_SECRET` — секрет для URL вебхуков фиатных шлюзов.

```bash
cd backend
npm install
cp .env.example .env   # впишите BOT_TOKEN, ADMIN_IDS и ключи платёжек
NODE_ENV=production node server.js
```

Проверка после старта: откройте `PUBLIC_URL/` (Mini App) и `/api/health`
(должен вернуть `{"ok":true}`). Незаполненные ключи сервер сообщает в лог при
старте — по ним видно, что ещё осталось вписать.

### 1. Настройте бэкенд

```bash
cd backend
npm install
cp .env.example .env   # заполните значения
node server.js
```

`server.js` автоматически подхватывает `backend/.env` (а если его нет —
корневой `.env`). Заглушки вроде `BOT_TOKEN=replace-me` и
`WEBHOOK_SECRET=replace-me` считаются незаполненными значениями.

Переменные окружения (`backend/.env.example`):

- `BOT_TOKEN` — токен бота из @BotFather (проверка подписи `initData` и Login Widget, без неё запросы отклоняются);
- `ADMIN_IDS` — Telegram ID гарантов/арбитров через запятую;
- `ADMIN_GROUP_ID` — необязательный `chat_id` основной админ-группы/супергруппы, куда бот отправляет служебные уведомления;
- `XROCKET_API_KEY` — ключ Rocket Pay: бот [@xRocket](https://t.me/xrocket) → **Rocket Pay → Create App → API Key**. Там же укажите webhook: `https://ваш-домен/webhook/xrocket`;
- `PGON_API_URL`, `PGON_API_KEY` — доступы шлюза PGon (поля запроса сверьте с документацией в `createPgonInvoice`);
- `NICEPAY_MERCHANT_ID`, `NICEPAY_SECRET` — доступы [NicePay](https://nicepay.io);
- `RUKASSA_SHOP_ID`, `RUKASSA_TOKEN` — доступы [RuKassa](https://lk.rukassa.pro);
- `WEBHOOK_SECRET` — секрет вебхуков шлюзов: добавьте его в URL вебхука в кабинете шлюза (`/webhook/rukassa?secret=...`);
- `BITPAPA_API_TOKEN` — токен API [Bitpapa](https://bitpapa.com) для автопроверки входящих переводов (пока переводы подтверждает гарант вручную через `POST /api/deals/:id/mark-paid`);
- `PUBLIC_URL` — публичный адрес бэкенда;
- `NODE_ENV=production` — боевой режим: локальная админ-зона без авторизации отключается (вернуть можно явным `LOCAL_ADMIN=1`);
- `TRUST_PROXY` — число прокси-хопов перед бэкендом (например `1` за nginx), иначе rate-limit и определение клиента работают по адресу прокси;
- `RATE_LIMIT_PER_MIN` — лимит запросов к `/api` и `/webhook` с одного IP в минуту (по умолчанию 300, `0` — отключить);
- `INITDATA_MAX_AGE` — максимальный возраст `initData` в секундах (по умолчанию 86400);
- `DATA_DIR` — каталог файлов данных (`deals.json`, `runtime_settings.json` и др.); в проде вынесите на постоянный том.

Тестовая среда xRocket: `XROCKET_API_URL=https://dev-pay.xrocket.tg` (бот @xrocket_dev_bot).

### 2. Подключите фронтенд к бэкенду

В `js/config.js`:

- `API_URL` — `"auto"`, если бэкенд отдаёт фронтенд сам (проверяется `/api/health`, при недоступности — демо-режим); `""` — принудительное демо; иначе полный адрес бэкенда;
- `BOT_USERNAME` — username бота для Telegram Login Widget (вход из браузера). Привяжите домен к боту: @BotFather → `/setdomain`;
- при необходимости — `FEE_PERCENT`, `SUPPORT_USERNAME`, `BITPAPA_ACCOUNT`, список `PAY_METHODS`.

### 3. Как проходит оплата

- **xRocket** — бэкенд создаёт счёт (`POST /tg-invoices` xRocket Pay API), Mini App открывает ссылку счёта, после оплаты xRocket шлёт webhook и сделка автоматически переходит в «Оплачена».
- **PGon / NicePay / RuKassa** — бэкенд создаёт платёж в шлюзе и возвращает ссылку на страницу оплаты; после оплаты шлюз шлёт webhook (`/webhook/pgon|nicepay|rukassa?secret=...`), сделка помечается оплаченной (сумма сверяется).
- **Bitpapa** — покупателю показываются реквизиты и код сделки для комментария к переводу; гарант сверяет поступление (вручную или через API Bitpapa в `verifyBitpapaTransfer`) и отмечает сделку оплаченной.

### 4. Выплата продавцу

Когда покупатель подтверждает приёмку, сделка становится «Завершена» — в этом месте
(`backend/server.js`, обработчик статуса `completed`) подключите выплату продавцу,
например переводом через xRocket (`POST /app/transfer`).

## API бэкенда

| Метод | Путь | Описание |
|---|---|---|
| GET | `/api/health` | Проверка доступности бэкенда |
| POST | `/api/auth/telegram` | Вход через Login Widget → токен сессии |
| GET | `/api/deals` | Сделки текущего пользователя |
| POST | `/api/deals` | Создать сделку |
| POST | `/api/deals/:id/status` | Сменить статус (переходы валидируются) |
| POST | `/api/deals/:id/invoice` | Создать счёт (xRocket / PGon / NicePay / RuKassa), вернуть ссылку |
| POST | `/api/deals/:id/bitpapa-claim` | Покупатель сообщил о переводе Bitpapa |
| POST | `/api/deals/:id/mark-paid` | Гарант подтвердил оплату (только `ADMIN_IDS`) |
| POST | `/webhook/xrocket` | Webhook xRocket об оплате счёта (проверка подписи) |
| POST | `/webhook/pgon` · `/webhook/nicepay` · `/webhook/rukassa` | Вебхуки шлюзов (`?secret=WEBHOOK_SECRET`, сверка суммы) |

## Telegram-бот (команды)

| Команда | Описание |
|---|---|
| `/start`, `/app` | Открыть Mini App, установить кнопку меню |
| `/post` | Пост-объявление с кнопками на шаблоны сделок продавца |
| `/group` | В группе: привязать текущую группу как основную админ-группу (только для `ADMIN_IDS`) |
| `/setup` | Перенастроить профиль и команды бота (только для `ADMIN_IDS`) |
| `/deal <ID>` | Проверить состояние сделки по ID |
| `/support` | Задать вопрос — следующее сообщение пересылается гарантам |
| `/admin` | Сводка + открытые задачи с кнопками действий (только для `ADMIN_IDS`) |

Бот работает в режиме long-polling и не требует публичного URL.  
Когда пользователь впервые обращается к боту, его `chat_id` сохраняется в `backend/user_chats.json`.
Это позволяет доставлять уведомления обеим сторонам сделки при изменении статуса.

Защищённые `/api/*`-запросы требуют заголовок `X-Telegram-Init-Data` (Mini App) или `X-Auth-Token` (после входа через Login Widget).

## Безопасность (модель доступа)

- Статика раздаётся по allowlist (`/`, `index.html`, `settings.html`, `css/`, `js/`) — файлы `backend/`, `.env`, `.git` и данные сделок наружу не отдаются.
- Переходы статусов ролевые: «исполнена» может отметить только продавец, «приёмка подтверждена» — только покупатель; счёт на оплату создаёт только покупатель. Оплату проставляют только вебхуки шлюзов или гарант.
- `initData` проверяется по подписи и возрасту (`INITDATA_MAX_AGE`), токены сессии — по HMAC и сроку действия.
- Вебхуки: xRocket — подпись `rocket-pay-signature` + тип события; шлюзы — `?secret=WEBHOOK_SECRET`, обязательная подпись при настроенных ключах, сверка суммы и валюты.
- Локальный админ-обход (`127.0.0.1` без авторизации) работает только вне `NODE_ENV=production` и не действует для запросов, прошедших через reverse-proxy.
- Автовыплата продавцу идемпотентна (`payoutId`) и запускается при подтверждении приёмки или решении спора в пользу продавца.

cd backend

## Важно

- В `backend/deals.json` — простое файловое хранилище для старта (запись атомарная); для продакшена перенесите на БД (PostgreSQL/SQLite).
- Комиссия и сумма к оплате всегда пересчитываются на сервере — клиенту не доверяем.
- Убедитесь, что деятельность гаранта соответствует законодательству вашей юрисдикции и правилам платёжных сервисов.
